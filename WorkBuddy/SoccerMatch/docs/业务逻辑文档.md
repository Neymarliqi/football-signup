## 二、用户身份判断逻辑

### 2.1 身份识别流程

```
app.onLaunch()
    │
    ├─ 同步读本地缓存 → 初始化 globalData（openid / userInfo / 用户头像缓存）
    │
    └─ 异步调用 getOpenid 云函数
         │
         ├─ 成功 → 存 openid + syncUserInfoFromCloud
         │         │
         │         ├─ users/{openid} 存在 → 写入 globalData + 写 user_registered
         │         └─ 不存在 → 用本地缓存（如果有）
         │
         └─ 失败 → 离线模式，读本地缓存

需要身份的操作（报名/待定/请假/发布/编辑资料/位置偏好）
    │
    └─ checkRegisterBeforeAction(action) / app.isUserRegistered()
         │
         ├─ 已注册 → 执行操作
         └─ 未注册 → 弹注册弹窗 → 用户关闭 → 取消操作
                    └─ 注册成功 → 自动继续执行原操作
```

### 2.2 身份判断方法

```js
// app.isUserRegistered() 逻辑：
// 1. 优先读本地永久标记 user_registered
// 2. 兼容旧版：本地 userInfo 有 nickName
// 3. 都没有 → 未注册
```

### 2.3 注册流程

```
用户触发需要身份的操作 → checkRegisterBeforeAction(action)
    │
    ├─ 已注册 → 直接执行操作
    └─ 未注册 → 弹注册弹窗
         │
         ├─ 用户关闭弹窗 → 取消操作，继续浏览
         │
         └─ 填写头像+昵称+勾选隐私 → 点击保存
              │
              └─ app.createUser(nickName, avatarBase64, positions)
                   │
                   ├─ wx.cloud.database().doc(openid).set() → 写入 users 集合
                   ├─ 更新 globalData.userInfo
                   ├─ wx.setStorageSync('userInfo', userInfo)
                   ├─ wx.setStorageSync('user_registered', true)  ← 永久标记
                   ├─ clearUserCache(openid)
                   └─ _executeRegisterCallbacks() → 通知各页面 onRegistered()
                        │
                        └─ 执行 _pendingAction（如弹出活动描述确认弹窗 / 直接执行待定/请假）
```

### 2.4 注册弹窗触发时机

| 触发场景 | 所在页面 | 触发方式 |
|----------|---------|---------|
| 点击报名 | detail.js | `checkRegisterBeforeAction('confirmed')` → 注册成功后弹出活动描述确认 |
| 点击待定/请假 | detail.js | `checkRegisterBeforeAction('pending'/'leave')` → 注册成功后直接执行 |
| TabBar 发布按钮 | custom-tab-bar | 已在首页时 `getCurrentPages()` 直接弹窗；非首页 `switchTab` + 全局标记 |
| 发布活动提交 | create.js | `submit()` 内检查注册 → 注册成功后自动提交 |
| 点击编辑资料 | profile.js | `checkRegisterBeforeAction('editProfile')` → 注册成功后跳转编辑页 |
| 点击位置偏好 | profile.js | `checkRegisterBeforeAction('selectPosition')` → 注册成功后继续选择 |

## 三、活动管理逻辑

### 3.1 活动状态

| 状态 | 值 | 说明 |
|------|------|------|
| 报名中 | open | 可报名/取消报名 |
| 进行中 | ongoing | 不可操作 |
| 已结束 | finished | 仅创建者可删除 |
| 已取消 | cancelled | 仅创建者可删除 |

### 3.2 权限矩阵

| 操作 | 创建者 | 已报名用户 | 普通用户 | 管理员 |
|------|--------|-----------|---------|--------|
| 查看活动 | ✅ | ✅ | ✅ | ✅ |
| 报名/取消 | ✅ | ✅ | ✅ | ✅ |
| 编辑活动 | ✅（open状态） | ❌ | ❌ | ❌ |
| 取消活动 | ✅（open状态） | ❌ | ❌ | ❌ |
| 删除活动 | ✅（finished/cancelled） | ❌ | ❌ | ❌ |

### 3.3 发布活动流程

```
create.js submit()
    │
    ├─ 检查必填字段（标题/时间/地点/人数）
    ├─ 注册检查：未注册 → 弹注册弹窗 → 注册成功自动继续
    ├─ 构建活动数据对象
    │
    └─ updateActivity 云函数（创建模式）
         │
         ├─ 首次创建 → db.collection('activities').add()
         └─ 编辑模式 → db.collection('activities').doc(id).update()
```

### 3.4 报名流程

```
detail.js onMainAction() → data-action="confirmed"
    │
    ├─ 注册检查：checkRegisterBeforeAction('confirmed')
    │   ├─ 未注册 → 弹注册弹窗 → 注册成功 → showConfirmModal()（活动描述确认）
    │   └─ 已注册 ↓
    │
    ├─ 设置 pendingAction = 'confirmed'
    └─ showConfirmModal()（3秒倒计时，需阅读完活动说明）
         │
         └─ confirmRegister() → updateRegistration 云函数
              │
              ├─ 首次报名 → 自动创建用户记录 + 更新活动 registrations
              ├─ 重复报名 → 更新状态
              └─ 返回最新活动数据 → 刷新页面
```

### 3.5 待定/请假流程

```
detail.js onSheetSelect() → data-action="pending"/"leave"
    │
    ├─ 关闭操作选择弹窗
    ├─ 注册检查：checkRegisterBeforeAction(action)
    │   ├─ 未注册 → 弹注册弹窗 → 注册成功 → doRegister(action) 直接执行
    │   └─ 已注册 ↓
    │
    └─ doRegister(action) → updateRegistration 云函数
```

## 四、头像管理逻辑

### 4.1 Base64 方案（v1.3.2+）

```
chooseAvatar → 临时路径
    │
    └─ app.uploadAvatar(tempPath)
         │
         ├─ 判断：http://tmp/ → 先 saveFileSync 持久化
         │
         └─ app.compressImageToBase64(filePath, 100)
              │
              ├─ wx.compressImage → 压缩到 100×100
              └─ wx.getFileSystemManager().readFile(base64) → data:image/jpeg;base64,xxx
```

### 4.2 头像显示

```js
// app.getDisplayAvatar(user) — 同步方法
// 1. 有 avatarBase64 → 直接用
// 2. 没有 → 降级用 avatarUrl（cloud://，可能显示不了）
// 3. 都没有 → 返回 defaultAvatar（/images/default-avatar.png）
```

### 4.3 头像同步

```
编辑资料页保存
    │
    ├─ 本地立即更新 globalData.userInfo + 用户缓存
    └─ 异步云端更新 users.doc(openid).update({ merge: true })
         │
         └─ clearUserCache → 下次其他页面 onShow 时静默刷新
```

## 五、缓存体系（v2.1.0）

### 5.1 用户信息缓存（三层）

```
fetchUsersWithCache(userIds, forceRefresh)
    │
    ├─ 第一层：内存缓存 usersCache[openid]
    │   ├─ 命中 → 直接返回（最快）
    │   └─ 未命中 ↓
    │
    ├─ 第二层：本地 Storage user_${openid}
    │   ├─ 命中 → 回填内存缓存 + 返回（冷启动可用）
    │   └─ 未命中 ↓
    │
    └─ 第三层：数据库查询 users 集合
        │
        ├─ 查到 → 写入内存 + Storage → 返回
        └─ 未查到 → 跳过该用户
```

### 5.2 Storage 数据结构

```js
// 用户缓存
wx.setStorageSync(`user_${openid}`, {
  nickName: '小明',
  avatarBase64: 'data:image/jpeg;base64,...',
  positions: [{value:'ST', order:1, label:'中锋'}],
  cachedAt: 1742960000000
})

// 活动列表缓存
wx.setStorageSync('activities_cache', {
  data: [...],          // 格式化后的活动数组
  timestamp: 1742960000000
})

// 注册标记（永久）
wx.setStorageSync('user_registered', true)

// 其他
wx.setStorageSync('openid', 'xxx')     // 用户身份
wx.setStorageSync('userInfo', {...})    // 完整用户数据
```

### 5.3 防膨胀清理

```
每次 saveUserToStorage 后触发 cleanupUserStorage()
    │
    ├─ 30天过期：cachedAt > 30天 → 自动删除
    └─ 8MB 硬上限：currentSize > 8MB → 按 cachedAt 排序删除最旧 20%
```

### 5.4 首页加载策略

```
onShow()
    │
    ├─ 30秒内返回 → silentRefreshUsers()（用缓存渲染，后台静默更新）
    ├─ 有数据+超30秒 → loadActivities(true, false)（不强制刷新用户缓存）
    └─ 无数据（冷启动）→ loadActivitiesFromCache() + loadActivities()（先渲染后刷新）
```

### 6.1 权限控制

| 操作 | 所有人 | 报名者+创建者 |

### 6.2 拖拽交互（纯拖拽，无点击操作）

```
touchstart → onPlayerDragStart → 记录起始位置
touchmove  → onPlayerDragMove  → 球员跟随手指

```

    │
    └─ tactics 集合

         └─ 无记录 → add

## 八、数据库字段规范

### 8.1 users 集合

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | = openid（使用 openid 作为文档 ID） |
| openid | string | 用户唯一标识 |
| nickName | string | 昵称 |
| avatarBase64 | string | base64 头像（v1.3.2+） |
| avatarUrl | string | 云存储头像（旧版，兼容） |
| positions | Array | 位置偏好 [{value, order, label}] |
| createdAt | Date | 创建时间 |
| updatedAt | Date | 更新时间 |

### 8.2 activities 集合

| 字段 | 类型 | 说明 |
|------|------|------|
| title | string | 活动标题 |
| status | string | open/ongoing/finished/cancelled |
| activityDate | Date | 活动日期 |
| time | string | 活动时间 |
| locationName | string | 地点名称 |
| location | object | 经纬度 {latitude, longitude, address} |
| matchType | string | 赛制 |
| maxPlayers | number | 最大人数 |
| fee | number | 费用 |
| fieldType | string | 场地类型 |
| notice | string | 比赛说明 |
| allowPending | boolean | 是否允许待定 |
| registrations | Array | 报名列表 [{openid, nickName, status, position}] |
| createdBy | string | 创建者 openid |
| createdAt | Date | 创建时间 |
| updatedAt | Date | 更新时间 |

### 8.3 关键规范

- **用户查询**: 必须用 openid，不能用 _id
- **位置存储**: 对象数组 `[{value, order, label}]`，按 order 排序显示首选/备选
- **effectiveStatus**: detail.js 和 create.js 中统一使用 effectiveStatus 判断（兼容日期计算）
- **UI 操作层次**: 主操作 → 功能操作 → 分隔线 → 管理操作
