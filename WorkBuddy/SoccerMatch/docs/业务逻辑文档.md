## 二、用户身份判断逻辑

### 2.1 身份识别流程

```
app.onLaunch()
    │
    ├─ 同步读本地缓存 → 初始化 globalData（openid / userInfo / 用户头像缓存）
    │
    └─ 异步调用 getOpenid 云函数
         │
         ├─ 成功 → 存 openid + syncUserInfoFromCloud
         │         │
         │         ├─ users/{openid} 存在 → 写入 globalData + 写 user_registered
         │         └─ 不存在 → 用本地缓存（如果有）
         │
         └─ 失败 → 离线模式，读本地缓存

需要身份的操作（报名/待定/请假/发布/编辑资料/位置偏好/加入球队）
    │
    └─ checkRegisterBeforeAction(action) / app.isUserRegistered()
         │
         ├─ 已注册 → 执行操作
         └─ 未注册 → 弹注册弹窗 → 用户关闭 → 取消操作
                    └─ 注册成功 → 自动继续执行原操作
```

### 2.2 身份判断方法

```js
// app.isUserRegistered() 逻辑：
// 1. 优先读本地永久标记 user_registered
// 2. 兼容旧版：本地 userInfo 有 nickName
// 3. 都没有 → 未注册
```

### 2.3 注册流程

```
用户触发需要身份的操作 → checkRegisterBeforeAction(action)
    │
    ├─ 已注册 → 直接执行操作
    └─ 未注册 → 弹注册弹窗
         │
         ├─ 用户关闭弹窗 → 取消操作，继续浏览
         │
         └─ 填写头像+昵称+勾选隐私 → 点击保存
              │
              └─ app.createUser(nickName, cloudPath, positions)
                   │
                   ├─ 头像上传 → wx.cloud.uploadFile → cloudPath
                   ├─ 写入 users 集合
                   ├─ 更新 globalData.userInfo
                   ├─ wx.setStorageSync('userInfo', userInfo)
                   ├─ wx.setStorageSync('user_registered', true)  ← 永久标记
                   ├─ clearUserCache(openid)
                   └─ _executeRegisterCallbacks() → 通知各页面 onRegistered()
```

### 2.4 注册弹窗触发时机

| 触发场景 | 所在页面 | 触发方式 |
|----------|---------|---------|
| 点击报名 | detail.js | `checkRegisterBeforeAction('confirmed')` → 注册成功后弹出活动描述确认 |
| 点击待定/请假 | detail.js | `checkRegisterBeforeAction('pending'/'leave')` → 注册成功后直接执行 |
| TabBar 发布按钮 | custom-tab-bar | 已在首页时 `getCurrentPages()` 直接弹窗；非首页 `switchTab` + 全局标记 |
| 发布活动提交 | create.js | `submit()` 内检查注册 → 注册成功后自动提交 |
| 点击编辑资料 | profile.js | `checkRegisterBeforeAction('editProfile')` → 注册成功后跳转编辑页 |
| 点击位置偏好 | profile.js | `checkRegisterBeforeAction('selectPosition')` → 注册成功后继续选择 |
| 申请加入球队 | team/home.js | `joinTeam()` 内检查注册 → 注册成功后自动申请 |

## 三、球队系统逻辑

### 3.1 球队创建流程

```
pages/team/create.js
    │
    ├─ 检查必填（Logo + 名称）
    ├─ 注册检查：未注册 → 弹注册弹窗 → 注册成功后继续
    ├─ 选择封面（可选）
    └─ createTeam 云函数
         │
         ├─ 写入 teams 集合
         ├─ 自动创建 team_members 记录（role: 'creator'）
         └─ 返回球队信息 → 跳转球队主页
```

### 3.2 球队身份判断

```
loadTeamInfo()
    │
    ├─ 调用 getMyTeams 云函数
    │   ├─ 查询 team_members 中该用户的记录
    │   ├─ 同时查询 teams 中 creatorOpenid 匹配
    │   └─ 合并确定身份（creator > admin > member）
    │
    └─ 返回 myRole:
        ├─ 'loading' → 加载中（三态判断，避免闪烁）
        ├─ 'creator' → 创建者（所有权限）
        ├─ 'admin' → 管理员（管理成员、处理申请、发布活动）
        ├─ 'member' → 普通成员
        └─ null → 非成员（显示申请按钮）
```

### 3.3 加入球队流程（审核模式）

```
非成员点击"申请加入"
    │
    ├─ 注册检查：未注册 → 弹注册弹窗
    │
    └─ joinTeam 云函数
         │
         ├─ 检查是否已是成员/创建者 → 是则拒绝
         ├─ 检查是否有 pending 申请 → 是则拒绝
         ├─ 写入 team_applications（status: 'pending'）
         └─ 返回结果 → 显示申请状态
```

### 3.4 申请审批流程

```
创建者/管理员在申请Tab点击"通过/拒绝"
    │
    └─ handleTeamApplication 云函数
         │
         ├─ 查询 team_applications（status: 'pending' + teamId + openid）
         │   ⚠️ 必须指定 status: 'pending'，否则查到旧的 rejected 记录会报错
         │
         ├─ 审批通过（action: 'approve'）
         │   ├─ 写入 team_members（role: 'member'）
         │   ├─ 更新 teams.memberCount +1
         │   └─ 删除 team_applications
         │
         └─ 审批拒绝（action: 'reject'）
              ├─ 更新 team_applications（status: 'rejected'）
              └─ 用户可重新申请
```

### 3.5 重新申请逻辑

```
用户被拒绝后重新申请
    │
    ├─ 检查 team_applications 中是否有 pending 申请 → 没有则允许
    └─ 写入新申请（覆盖旧记录或新建）
```

### 3.6 球队分享与访问

```
分享卡片进入 /pages/team/home?teamId=xxx
    │
    ├─ loadTeamInfo() → 获取球队信息
    ├─ loadMyRole() → 调用 getMyTeams 判断身份
    └─ 根据身份显示对应界面
        ├─ 加载中 → 显示loading状态（三态判断）
        ├─ 成员/创建者 → 正常显示
        └─ 非成员 → 显示申请按钮（审核模式）或直接加入（公开模式）
```

## 四、活动管理逻辑

### 4.1 活动状态

| 状态 | 值 | 说明 |
|------|------|------|
| 报名中 | open | 可报名/取消报名 |
| 进行中 | ongoing | 不可操作 |
| 已结束 | finished | 仅创建者可删除 |
| 已取消 | cancelled | 仅创建者可删除 |

### 4.2 权限矩阵

| 操作 | 创建者 | 已报名用户 | 普通用户 | 管理员 |
|------|--------|-----------|---------|--------|
| 查看活动 | ✅ | ✅ | ✅ | ✅ |
| 报名/取消 | ✅ | ✅ | ✅ | ✅ |
| 编辑活动 | ✅（open状态） | ❌ | ❌ | ❌ |
| 取消活动 | ✅（open状态） | ❌ | ❌ | ❌ |
| 删除活动 | ✅（finished/cancelled） | ❌ | ❌ | ❌ |

### 4.3 发布活动流程

```
create.js submit()
    │
    ├─ 检查必填字段（标题/时间/地点/人数）
    ├─ 注册检查：未注册 → 弹注册弹窗 → 注册成功自动继续
    ├─ 选择球队（可选）
    │   ├─ 选择球队 → visibility = 'team'，teamId/teamName 填充
    │   └─ 不选择 → visibility = 'public'
    └─ updateActivity 云函数（创建模式）
         │
         ├─ 首次创建 → db.collection('activities').add()
         └─ 编辑模式 → db.collection('activities').doc(id).update()
```

### 4.4 报名流程

```
detail.js onMainAction() → data-action="confirmed"
    │
    ├─ 注册检查：checkRegisterBeforeAction('confirmed')
    │   ├─ 未注册 → 弹注册弹窗 → 注册成功 → showConfirmModal()（活动描述确认）
    │   └─ 已注册 ↓
    │
    ├─ 设置 pendingAction = 'confirmed'
    └─ showConfirmModal()（3秒倒计时，需阅读完活动说明）
         │
         └─ confirmRegister() → updateRegistration 云函数
              │
              ├─ 首次报名 → 自动创建用户记录 + 更新活动 registrations
              ├─ 重复报名 → 更新状态
              └─ 返回最新活动数据 → 刷新页面
```

### 4.5 待定/请假流程

```
detail.js onSheetSelect() → data-action="pending"/"leave"
    │
    ├─ 关闭操作选择弹窗
    ├─ 注册检查：checkRegisterBeforeAction(action)
    │   ├─ 未注册 → 弹注册弹窗 → 注册成功 → doRegister(action) 直接执行
    │   └─ 已注册 ↓
    │
    └─ doRegister(action) → updateRegistration 云函数
```

## 五、头像管理逻辑（v3.0.0 云存储方案）

### 5.1 头像上传流程

```
chooseAvatar → 临时路径
    │
    └─ app.uploadAvatar(tempPath)
         │
         ├─ 判断：http://tmp/ → 先 saveFileSync 持久化
         ├─ 压缩图片 → 200×200，quality: 70
         ├─ wx.cloud.uploadFile → cloudPath（完整 fileID）
         └─ 返回 cloudPath（如 cloud://cloud1-xxx/avatars/xxx.jpg）
```

### 5.2 头像显示

```js
// app.getDisplayAvatar(user) — 同步方法
// 优先级：
// 1. displayAvatar（fetchUsersWithCache 已预解析的临时 URL）
// 2. avatarBase64（旧数据兼容）
// 3. cloudPath → _resolveCloudPath（异步获取后缓存）
// 4. defaultAvatar（/images/default-avatar.png）
```

### 5.3 头像同步

```
编辑资料页保存
    │
    ├─ 本地立即更新 globalData.userInfo + 用户缓存
    └─ 异步云端更新
         ├─ 新头像 → uploadAvatar → cloudPath
         ├─ users.doc(openid).update({ merge: true })
         └─ clearUserCache → 下次其他页面 onShow 时静默刷新
```

### 5.4 关键教训（踩坑记录）

- `wx.cloud.uploadFile` 返回**完整 fileID**：`cloud://cloud1-5gbn5i1p97239e9d.636c-xxx/avatars/xxx.jpg`
- 存入数据库必须存**完整 fileID**，不能只存相对路径
- `wx.cloud.getTempFileURL` 的 fileList 参数也必须用**完整 fileID**
- `<image>` 组件可以直接使用 `cloud://` 格式，微信会自动转换
- **最佳实践**：数据库存完整 fileID → fetchUsersWithCache 批量 getTempFileURL → 存 displayAvatar → image 组件用 displayAvatar

## 六、缓存体系（v3.0.1）

### 6.1 用户信息缓存（三层）

```
fetchUsersWithCache(userIds, forceRefresh)
    │
    ├─ 第一层：内存缓存 usersCache[openid]
    │   ├─ 命中 → 直接返回（最快）
    │   └─ 未命中 ↓
    │
    ├─ 第二层：本地 Storage user_${openid}
    │   ├─ 命中 → 回填内存缓存 + 返回（冷启动可用）
    │   └─ 未命中 ↓
    │
    └─ 第三层：数据库查询 users 集合
        │
        ├─ 查到 → 写入内存 + Storage → 返回
        └─ 未查到 → 跳过该用户
```

### 6.2 Storage 数据结构

```js
// 用户缓存
wx.setStorageSync(`user_${openid}`, {
  nickName: '小明',
  cloudPath: 'cloud://cloud1-xxx/avatars/xxx.jpg',
  avatarBase64: 'data:image/jpeg;base64,...',  // 旧数据
  positions: [{value:'ST', order:1, label:'中锋'}],
  cachedAt: 1742960000000
})

// 注册标记（永久）
wx.setStorageSync('user_registered', true)

// 其他
wx.setStorageSync('openid', 'xxx')
wx.setStorageSync('userInfo', {...})
```

### 6.3 防膨胀清理

```
每次 saveUserToStorage 后触发 cleanupUserStorage()
    │
    ├─ 30天过期：cachedAt > 30天 → 自动删除
    └─ 8MB 硬上限：currentSize > 8MB → 按 cachedAt 排序删除最旧 20%
```

### 6.4 首页加载策略

```
onShow()
    │
    ├─ 30秒内返回 → silentRefreshUsers()（用缓存渲染，后台静默更新）
    ├─ 有数据+超30秒 → loadActivities(true, false)
    └─ 无数据（冷启动）→ loadActivitiesFromCache() + loadActivities()
```

## 十一、三态判断模式

为避免页面闪烁，采用三态判断模式：

```js
// home.js
data: {
  myRole: 'loading'  // 初始值设为 loading
}

// home.wxml
<view wx:if="{{myRole === 'loading'}}" class="loading-bar">
  <text class="loading-text">加载中...</text>
</view>

<view wx:elif="{{myRole === null}}" class="non-member-bar">
  <!-- 申请按钮 -->
</view>

<view wx:else>
  <!-- 成员界面 -->
</view>
```

适用场景：
- 球队主页（team/home）
- 需要异步加载身份的页面

## 十二、常见 Bug 与修复

### Bug 1: 重新申请后审批显示"已经操作过了"

**原因**: `handleTeamApplication` 查询申请记录时没有指定 `status: 'pending'`，查到了之前的 `rejected` 记录

**修复**: 在 where 条件中添加 `status: 'pending'`

```js
// ❌ 错误
.where({ teamId, openid: targetOpenid }).get()

// ✅ 正确
.where({ teamId, openid: targetOpenid, status: 'pending' }).get()
```

### Bug 2: 球队主页申请按钮闪烁

**原因**: `myRole` 初始值为 `null`，在异步加载完成前会显示申请按钮

**修复**: 初始值改为 `'loading'`，加载完成后再判断身份
